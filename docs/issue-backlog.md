# Blackbox Flight Analyzer Issue Backlog

このバックログは「見た目を整える」ためではなく、MVP を現実のログと DVR に耐えさせるための実行順です。
順番を崩すと、薄い改善が増えるだけでプロダクトは強くなりません。

## 実行ルール

- 新機能追加より先に、実データ耐性と判断精度を固める
- UI 改善は、観測性・耐障害性・正規化の後にやる
- 各 issue は完了条件を満たすまで閉じない
- 「なんとなく改善した」は完了ではない。再現可能な確認結果が必要

## Issue 1: 実ログ / 実 DVR 検証セットを固定する

- 優先度: P0
- 狙い: 開発者の都合のいい入力だけで判断しない状態を作る
- 背景: 今の最大のリスクはコード不足ではなく、現実の入力差に対する無知

### 作業

- 検証用データセットの配置方針を決める
- 少なくとも次のケースを揃える
- 正常に読める `.bbl`
- 未知ヘッダ警告が出る `.bbl`
- AUX / RPM の欠損または差異がある `.bbl`
- `ARMED` OCR が通る DVR
- `ARMED` OCR が外れる DVR
- 各サンプルに観察メモを付ける

### 完了条件

- データセット一覧と期待観察結果が文書化されている
- どのファイルが何の検証用かチームが迷わない
- 以後の issue が同じ入力セットで再検証できる

## Issue 2: selector の欠損時契約を明文化し実装に反映する

- 優先度: P0
- 対象: [src/domain/blackbox/selectors/flightSelectors.js](/E:/src/blackbox-flight-analyzer/src/domain/blackbox/selectors/flightSelectors.js)
- 狙い: 上位レイヤーが欠損データに振り回されない状態を作る
- 背景: 今は `0` と `null` の意味が混ざっていて、欠損と実測値ゼロが区別しきれていない

### 作業

- 取得不能な値の表現を決める
- `rc` `rcRaw` `setpoint` `gyro` `error` `motors` `rpm` `aux` の契約を整理する
- 候補フィールド解決の優先順位を明文化する
- 欠損時フォールバックが UI とイベント判定に与える影響を洗い出す
- 必要なら selector 補助関数を分離する

### 完了条件

- 欠損値の扱いがコードと文書で一致している
- UI が欠損を誤って正常値として表示しない
- イベント判定が欠損データで暴れにくくなる

## Issue 3: パーサ失敗と OCR 失敗を局所化する

- 優先度: P0
- 対象: [src/app/App.jsx](/E:/src/blackbox-flight-analyzer/src/app/App.jsx)
- 狙い: 一部の失敗でアプリ全体が死なない状態を作る
- 背景: 失敗を正常系として扱えていない。これは製品として弱い

### 作業

- React error boundary を追加する
- ログ読み込み失敗時の表示を対象ファイル単位にする
- OCR auto sync 失敗時の状態表示を分かるようにする
- 失敗後も手動 offset 調整とログ閲覧を継続できるようにする

### 完了条件

- 壊れたログや OCR 失敗で全体 UI が blank にならない
- ユーザーが失敗内容と次の操作を理解できる
- 失敗した後でも他の flight を継続して触れる

## Issue 5: イベント定義をレビュー価値基準で再設計する

## Issue 6: compare 指標を揃った条件比較に寄せる

- 優先度: P1
- 対象: [src/domain/compare/compareMetrics.js](/E:/src/blackbox-flight-analyzer/src/domain/compare/compareMetrics.js)

## Issue 7: OCR auto sync の成功率より失敗時復帰を優先して改善する

## Issue 8: UI の情報密度を検証結果に合わせて削る

- 「出せるから出している」要素が減っている
